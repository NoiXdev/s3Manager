# i18n for s3manager ŌĆö Design

**Date:** 2026-06-11
**Status:** Approved (pending spec review)
**Languages:** English (source), German, French, Polish, Dutch, Romanian

## Goal

Make the renderer UI fully translatable and ship machine-translated baselines for
German (`de`), French (`fr`), Polish (`pl`), Dutch (`nl`), and Romanian (`ro`),
with English (`en`) as the source language. Users get the app in their OS language
on first run, with an in-app override in Settings.

## Decisions

| Question | Decision |
| --- | --- |
| Library | `i18next` + `react-i18next` |
| Catalog loading | Bundled at build time (all 6 locales in `resources`), synchronous init |
| Default language | OS locale on first run, fallback English |
| Override | Language dropdown in Settings, persisted via `SettingsRepo` (mirrors `theme`) |
| Translation source | Machine-translated full baselines for de/fr/pl/nl/ro |
| Scope | Renderer UI only; main-process/AWS error strings stay English |

## Architecture

```
src/renderer/i18n/
  index.ts                 # initialises i18next with bundled resources, fallbackLng 'en'
  locales/
    en.json  de.json  fr.json  pl.json  nl.json  ro.json
```

- **Init module** (`i18n/index.ts`): calls `i18n.use(initReactI18next).init({ resources, lng: 'en', fallbackLng: 'en', interpolation: { escapeValue: false } })`. Imported once (side-effect) from `main.tsx` before `<App />` renders so the first paint is localized.
- **Single `translation` namespace** per locale. Keys nested by feature:
  `common.*`, `nav.*`, `settings.*`, `files.*`, `cors.*`, `objectLock.*`,
  `sync.*`, `accounts.*`, `connections.*`, `dashboard.*`.
- **Dynamic values** use i18next interpolation (`{{count}}`, `{{name}}`) and
  plural keys (`key_one` / `key_other`) where a string embeds a count.
- **English values equal today's exact strings**, so existing tests that match on
  English text continue to pass without change.

## Language selection (mirrors `theme`)

### Settings model ŌĆö `src/main/settings/appSettings.ts`
- Add `language: LanguagePreference` to `AppSettings`, where
  `LanguagePreference = 'system' | 'en' | 'de' | 'fr' | 'pl' | 'nl' | 'ro'`.
- Default `'system'`.
- `readSettings` reads/validates the stored `language` (fallback `'system'` if
  missing or not a known value).
- `writeSettings` validates and persists `language` when present in the patch.
- No new IPC channel: `getSettings`/`setSettings` already carry the whole settings
  object. (One `npm start` restart is needed because `appSettings.ts` runs in the
  main process ŌĆö see `main-process-restart-required` note.)

### Resolution hook ŌĆö `src/renderer/hooks/useLanguage.ts`
Sibling to `useTheme`. Signature: `useLanguage(preference: LanguagePreference | undefined): void`.
- When preference is `'system'` (or undefined): resolve from `navigator.language`
  (reflects OS locale in Electron) ŌåÆ nearest supported locale by primary subtag
  (e.g. `de-DE` ŌåÆ `de`) ŌåÆ `'en'` if unsupported.
- When preference is an explicit locale: use it directly.
- Calls `i18n.changeLanguage(resolved)` in an effect keyed on `preference`.
- Wired in `App.tsx` next to `useTheme(settings.data?.theme)`:
  `useLanguage(settings.data?.language)`.

### Settings UI ŌĆö `src/renderer/components/settings/SettingsScreen.tsx`
- Add a **Language** `<select>` above or below Appearance, same styling.
- Options: `System` plus each locale labelled in its own language ŌĆö
  English, Deutsch, Fran├¦ais, Polski, Nederlands, Rom├ón─ā.
- `onChange` calls `save.mutateAsync({ language })` and shows the existing
  "Settings saved" toast (itself translated).

## Migration

Replace hardcoded JSX strings with `t('key')` across the renderer (~40 components),
including:
- `SectionNav` ŌĆö `PRIMARY`/`SECONDARY` labels become keys, resolved via `useTranslation`.
- `App.tsx` ŌĆö sidebar title, "Manage connections", "Coming soon", footer.
- All dialogs, buttons, labels, empty states, and renderer-triggered toast text.

Main-process / S3 / AWS error strings are **not** translated in this pass.

## Testing

- Initialize i18next synchronously in the vitest setup so `useTranslation` resolves
  real strings in component tests.
- Existing English-text assertions remain valid because the `en` catalog mirrors
  current strings exactly.
- New tests:
  - `useLanguage`: `'system'` ŌåÆ `navigator.language` mapping, primary-subtag match,
    fallback to `en` for unsupported locales, explicit-locale passthrough.
  - Settings Language dropdown: renders options, persists selection via `save`.

## Out of scope (this pass)

- Main-process / AWS error message translation.
- Date/number/currency localization beyond what individual strings require.
- RTL layout (none of the six languages are RTL).
- Lazy-loading catalogs per locale.

## Domain terms kept untranslated in all locales

bucket, CORS, S3, ACL, presign, endpoint, sync.
